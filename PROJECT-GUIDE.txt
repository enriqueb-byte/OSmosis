WHAT THE APP DOES
-----------------
Call Gym is a practice drill for people who support or sell Jobber (field service
software). A user runs a short session: they see 10 random scenario questions
(customer-style queries), answer each in bullet points under a time limit, then
get a summary. They can copy a "transcript" (questions + their answers) and
paste it into Gemini to get a scorecard and feedback. No login, no backend—
everything runs in the browser.


HOW TO RUN THE APP LOCALLY
--------------------------

What is the terminal?
  The terminal is a window where you type text commands instead of clicking.
  Commands run in the "current folder" (directory). For npm run dev to work,
  the terminal must be "in" your Call Gym project folder—the one that contains
  package.json, src/, index.html, etc.

Do I need to navigate somewhere before running "npm run dev"?
  Yes. The terminal has to be in the Call Gym project folder. If you open the
  terminal from Cursor with the Call Gym folder open as your project, it usually
  starts there already. If not, you need to "go to" that folder first (see
  below).

Steps to run the app
  1. Open the Cursor terminal (not the Mac Terminal app):
       Terminal → New Terminal, or press Ctrl+` / Cmd+`.
       It opens in your project folder so you can run commands right away.
  2. If you ever use the Mac Terminal app instead, go to the project folder first by typing (then press Enter):
       cd "/Users/enrique.b/Documents/AI Learning/Call Gym"
       (Use your actual path if the project lives somewhere else. You can
        drag the project folder into the terminal window to paste its path.)
  3. Run the dev server:
       npm run dev
  4. Open the URL it prints (e.g. http://localhost:5173) in your browser.

Do NOT try to open index.html by double-clicking it—that won't work. You must
use the dev server and the URL it gives you.

How to RESTART the dev server
  • In the terminal where "npm run dev" is running, press:  Ctrl + C
    (Mac: Ctrl + C, or sometimes Cmd + C depending on terminal.)
  • That stops the server. Then run again:  npm run dev
  • Open the new URL it prints (e.g. http://localhost:5173 or 5174) in your
    browser. Bookmark that URL for quick access.

How to check that the transcript includes the preamble (instructions for Gemini)
  1. Start the app: in terminal run "npm run dev", then open the URL in your
     browser.
  2. Do a practice session: click "Start session", answer a scenario or two (you
     can type anything), click Next until you see "Session complete."
  3. Click the "Copy transcript" button.
  4. Open any text editor (Notepad, TextEdit, Notes, or paste into a new
     Cursor file). Paste (Ctrl+V or Cmd+V).
  5. At the top you should see:
       # Call Gym Session Transcript

       ## Instructions for the scorer (e.g. Gemini)

       You are scoring a Call Gym practice session...
       (the full text from gemini-preamble.md)

       ---

       ## Session metadata
       ...
       ## Scenario Q&As (score these)
       ...
     If you see that, the preamble is included. You can then paste the same
     text into Gemini to get a scorecard.

To build the app for deployment (e.g. GitHub Pages):
   npm run build
Then the built files are in the "dist" folder.


FOLDER AND FILE ORGANIZATION
----------------------------

  Project root
  ------------
  index.html          The single HTML page; points the browser to the app.
  package.json        Lists dependencies and scripts (dev, build, preview).
  vite.config.js      Build tool config (e.g. base path). Rarely needs changes.
  tailwind.config.js  Styling (Tailwind) config. Rarely needs changes.
  .cursorrules        Rules for Cursor (the IDE) when editing this codebase.
  PROJECT-GUIDE.txt   This file.

  src/
  -----
  main.jsx            Entry point: loads the main App and global styles.
  App.jsx             Main app logic: session flow (start → scenarios → summary),
                      how many scenarios per session (10), timer length (60 sec).
  index.css           Global styles (Tailwind + a few custom classes).

  src/components/
  -----------------
  StartScreen.jsx     The first screen: filter checkboxes (topic, difficulty, etc.)
                      and "Start session" button.
  ScenarioCard.jsx   One scenario: shows the query, timer, and text area for
                      the user's response; handles submit and next.
  SessionSummary.jsx  End screen: lists scenarios answered, "Copy transcript"
                      button. The transcript includes instructions for Gemini
                      so the user can paste once and get a scorecard.

  src/data/
  ----------
  scenarios.json      All practice scenarios (see "Scenario schema" below).
  scenarioFilters.js  Defines filter criteria and options (Topics,
                      Difficulty, Customer type). Edit to add/rename
                      options; scenarios must use the same values.
  gemini-preamble.md  The instructions that get pasted into the transcript for
                      Gemini. Edit this to change how Gemini scores and what
                      kind of scorecard the user gets (tone, format, criteria).

PRACTICE VS ASSESSMENT MODE
---------------------------
  On the start screen you can choose Practice or Assessment.
  • Practice: You pick how many questions, time per question, and optional
    filters (Topics, Difficulty, Customer type). Full flexibility.
  • Assessment: Fixed session—number of questions, time, and filters come from
    src/data/testConfig.js (e.g. 10 questions, 60 sec each, all topics).
    Change TEST_QUESTION_COUNT, TEST_TIMER_SECONDS, or TEST_FILTERS there
    to customize the default assessment.

  The "How it works" section at the bottom of the start screen explains
  where scenarios come from, how the transcript and preamble work, how
  scoring is done (in Gemini, not in the app), and who built it. Edit the
  text in StartScreen.jsx inside the CollapsibleSection "How it works" if
  you want to change that copy (e.g. "Built by you" → your name).


BASIC CHANGES YOU MIGHT WANT TO MAKE
------------------------------------
  • Add or edit practice scenarios    →  Edit src/data/scenarios.json
  • Change how the scorecard looks    →  Edit src/data/gemini-preamble.md
  • Change Assessment mode defaults   →  Edit src/data/testConfig.js
  • Change number of scenarios (e.g. 5) or timer (e.g. 90 sec) for Practice
    →  Edit src/App.jsx (defaults used when no options passed)
  • Change app title or start-screen text  →  Edit index.html (title),
    →  Edit src/components/StartScreen.jsx (heading and subtitle)
  • Change styling (colors, spacing)  →  Edit src/index.css and/or
    →  Edit the .jsx files (they use Tailwind classes like bg-indigo-500)
  • Add or change filter options (e.g. new topic name)  →  src/data/scenarioFilters.js


SCENARIO SCHEMA (FOR YOUR SCENARIOS)
---------------------------------------
Each scenario in scenarios.json must have: id, query. For filtering, use
these field names and values exactly:

  topic      Topic name: "Leads" | "Quotes" | "Scheduling" | "Operations" |
             "Communication" | "Invoicing & Payments" | "Finance" | "App Integrations"
  difficulty 1 (Basics), 2 (Workflow), or 3 (Strategy/ROI)
  persona   "busy_bee" | "micro_manager" | "burned_pro" | "tech_rookie"

  Optional (metadata only; not used for filtering):
  tier       "connect" or "grow" — for authoring when scenario is plan-specific.

Example:
  {
    "id": "L-01",
    "topic": "Leads",
    "tier": "connect",
    "difficulty": 1,
    "persona": "busy_bee",
    "query": "I'm losing leads because I can't answer the phone..."
  }

Optional: title, objective, ideal_keywords (for display or Gemini). ScenarioCard
shows id, query, and level (difficulty) if present. Scenarios missing filter
fields still appear when "any" is selected; they are skipped when filters
are applied.


DEPLOYMENT (GITHUB PAGES)
-------------------------
The app is set up to deploy to GitHub Pages on push to the main branch. The
workflow is in .github/workflows/deploy-pages.yml. After a push to main, the
"dist" folder from "npm run build" is published. The live site is the one that
works when opened in a browser; for local work, always use "npm run dev" and
the URL it prints.
